Highlights overview — see every highlight, grouped by color label

Today highlights live only inside the PDF and chat text. These three variants add a way to review them all at once, filtered by color label ("Important", "Agree", "Reference", "Question", "Disagree"). Variant A is a persistent panel, Variant B a focus mode inside the PDF, Variant C a full-screen board.

Variant A — chosen (final) Highlights drawer over the chat column

Decided 2026-07-21: the panel is a drawer that slides over the chat column (~200 ms), opened from a highlighter icon in the now-permanent chat header. The total count appears exactly once — in the "All" chip; no icon badge, no extra counter. The drawer lists every highlight of the whole chat tree — PDF and chat highlights — grouped by color label in document order (PDF pages first, then chat highlights in tree order). Chips multi-select. Clicking a card scrolls precisely to the highlight and flashes it; clicking a chat card switches the branch and closes the drawer.

3. Method

We introduce Coordinated Boltzmann Monte Carlo Tree Search (CB-MCTS), which replaces the deterministic UCT rule with a stochastic Boltzmann policy and incorporates a decaying entropy-based bonus for sustained yet focused exploration.

Coordination among agents is achieved through a marginal contribution function that aligns each agent's local decisions with the global objective. In contrast to prior work, we claim that no communication between agents is required during rollout, since the contribution function is evaluated only at the root.

Empirically, the temperature schedule follows an inverse-square-root decay, which matches the regret bound of UCT up to logarithmic factors while improving robustness under partial observability.

The highlighter icon in the chat header toggles the drawer (esc closes; state not persisted across sessions). The total shows only in the "All" chip. Chips multi-select — e.g. "Question" + "Disagree" together; "All" resets; empty groups are hidden. Clicking a card scrolls the PDF to the exact highlight rect (or switches branch, closes the drawer and scrolls the chat message) and flashes the highlight ~1.5 s. Right-click on a card opens the same highlight actions menu as in the document (recolor, delete, open linked chat). Trees without a PDF work identically — the list then holds only chat highlights.

Variant B Review mode — filter inside the PDF

No extra column. A floating review bar appears above the document with the same label chips; selecting "Disagree" dims every other highlight in place, so the matching passages pop out in their real context. Arrows (or n/p) jump between matches; a margin minimap shows where matches sit across all pages.

Comparison & recommendation
VariantStrengthWeaknessBest for
A — Panel Always at hand while reading; context stays visible; cheap to build (right column already exists for chats). Little room per card; long quotes truncate. Everyday use while working in a tree.
B — Review mode Highlights appear in their real context — no mental mapping back to the page. Only one label at a time feels natural; no cross-page overview beyond the minimap. Re-reading a paper with one question in mind ("where did I disagree?").
C — Board Complete overview; relabeling via drag & drop; room for notes and branch links. Leaves the reading context; a full extra view to build and maintain. Final review pass / exporting takeaways.

Decision (2026-07-21, grill session): Variant A chosen and finalized. Scope = whole chat tree; document order (PDF pages first, then chats in tree order); multi-select chips; drawer over the chat column, opened from a now-permanent chat header (highlighter icon, no badge — the total appears only in the "All" chip); data via new GET /api/chats/:rootId/tree-highlights (recursive CTE + UNION); shared store following the useLabels pattern (CRUD hooks invalidate, panel always in sync); precise scroll with ~1.5 s flash (new scrollToHighlight in PdfView, scrollToMessage in ChatArea); right-click on a card reuses HighlightActionsMenu. Color labels stay global (existing highlight_labels table): renaming a label in any right-click popup updates chips, group headers and popups everywhere. Variants B and C remain future options.

Correction (2026-07-21, after first build): the header button carries a visible "Highlights" text label next to the icon (icon-only was too easy to miss), and the jump flash is a pulsing glow in the highlight's own deep color (2 × 0.75 s) instead of a black ring — implemented as a separate overlay so the multiply-blended mark itself stays untouched. Multi-line highlights glow as one silhouette: all line rects sit in a single container whose drop-shadow follows their union — no brighter seams where per-line shadows would overlap.

Correction (2026-07-21, second pass): the double pulse stays (a single-envelope variant was tried and rejected — the two beats are what makes the jump noticeable), but the glow itself is tighter and quieter: a thin 1px contour plus one soft 8px ambient shadow (was 5px + 12px neon), and the color wash over the text dropped from 28% to 16%.
